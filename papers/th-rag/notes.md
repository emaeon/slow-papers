# TH-RAG : Topic-Based Hierarchical Knowledge Graphs for Robust Multi-hop Reasoning in Graph-based RAG Systems

## 이 논문이 해결하려고 하는 문제

트리플렛 기반 그래프의 파편화된 정보와 낮은 의미적 연결성으로 인해 추론 시 의미를 왜곡하는 문제

## 기존 방법의 한계

llm이 만든 트리플렛 기반 그래프는 의미적 연결성이 낮고 파편화되어있어서, 일관성을 감소시키고 추론을 방해한다. 기존 연구들( 클러스터링하거나 커뮤니티 디텍션, 근사 그래프 알고리즘들)은 연결성을 복구하기를 시도했으나, 높은 계산 비용을 초래하고, 의미를 왜곡할 위험이 있다.

말뭉치에서 트리플을 추출해서 직접 지식그래프를 추출하는 방법이 이어지고 있다. 이런 세분화된 표현은 정보를 의미적 수준에서 구조화함으로써 추론의 정확성을 높인다. 이러한 방식은 청크안에 있는 트리플들이 충분히 연결되어있다고 가정한다. 하지만 현실의 LLM 추출 결과는 그렇지 않다.
현실에서는 LLM이 생성한 트리플렛 기반 지식그래프가 자주 파편화가 되며, 서로 제대로 연결되지않은 고립 하위 그래프들이 생긴다 .
이러한 파편화는 그래프의 전반적인 의미적 일관성을 떨어뜨릴 뿐만 아니라 효과적인 멀티홉 추론도 방해한다 엔터티 사이의 핵심적인 관계 경로가 누락되어있거나 끊어져있을 수가 있기 때문이다.
그 결과 모델은 여러 곳에 흩어진 정보를 통합하는데 어려움을 겪고, 이는 궁극적으로 정확하면서 문맥에 근거한 답변을 생성하는 능력을 저해한다.
(트리플 기반 그래프는 충분히 연결되어있다고 가정 -> 실제로는 파편화 되어있음 -> 핵심 관계 경로가 끊김 -> 모델이 흩어진 정보를 통합못함 -> 정확한 답변 생성이 어려워짐)

이러한 문제를 해결하기 위해 그래프 보강 기술들을 기존 연구에서 제안함.
예를 들어, 커뮤니티 디텍션, 클러스터링 기반 요약, hnsw와 같은 알고리즘

그러나 이러한 방법들은 심각한 드로우백(단점)을 겪음:
예를 들어 추가적인 계산 비용의 발생 또는 세부적인 의미 관계를 왜곡시킨다,
이 두문제는 검색된 정보의 신뢰성과 정확성을 모두 저해한다.

## 제안 방법

트리플렛을 서브토픽과 토픽으로 조직화 해서 연결성을 향상시키고, 분산된 정보를 통합하며, 안정적인 멀티홉 추론을 지원한다.
트리플렛, 서브토픽, 토픽으로 구성된 3개 층으로 구성된 지식 그래프 구축 프레임워크를 제안한다.
이러한 의미적 계층 구조는 그래프 연결성을 높이고, 파편화된 정보들을 통합하기 쉽게 하며, 효율적인 멀티홉 추론과 말뭉치에 대한 전반적인 이해를 지원한다.

1. Hierarchical Graph Construction
   LLM이 트리플렛, 서브토픽, 토픽을 동시에 추출해서 의미적으로 구조화된 그래프를 만든다.
   트리플렛 추출하고 서브토픽과 토픽 주석을 추가해 계층적으로 구조화된 그래프를 만든다.
   각 엔터티는 하나 이상의 서브토픽에 연결되고, 각 서브토픽은 하나 이상의 토픽에 연결된다. 이로써 의미적 포함 관계를 나타내는 계층 구조가 만들어진다.
   엔터티: 사실 단위를 표현
   서브토픽: 의미적으로 관련있는 엔터티들을 묶음
   토픽:서브토픽 그룹을 더 상위 범주로 추상화
   의미적 근거성을 보장하기 위해, llm에게 말뭉치의 실제 내용과 관련된 서브토픽과 토픽만 추출하도록 지시
   TH-RAG의 노드는 Entity·Subtopic·Topic 세 종류이고, 엣지도 Entity 관계·Subtopic–Entity 연결·Topic–Subtopic 연결 세 종류다.
   각 트리플 엣지는 문장 수준의 검색을 위해 해당 트리플의 원문 문장을 속성으로 저장한다. 이 문장들은 말뭉치에서 직접 가져온 것이므로 환각 위험을 최소화할 수 있다.
   각 엔터티에 실제 사실 관계 엣지뿐 아니라 의미적 계층 엣지도 추가하여, 직접 관계가 끊겨도 완전히 고립될 가능성을 낮춘다.
   '''핵심: 트리플 관계가 끊겨도, 토픽/서브토픽을 통한 우회 연결을 제공해 원문 문장을 보존하여 근거 검색에 사용한다'''

2. Topic-based Graph Traversal
   질문이 들어오면 전체 엔터티를 바로 검색하지 않고 두 단계로 범위를 좁힌다.
   2.1 Topic 선택
   질문 + 전체 토픽 후보 목록 -> llm -> 관련 topic n_t 개

2.2 subTopic 선택
질문 + 선택된 Topic의 Subtopic 후보 -> llm -> 관련 Subtopic N_ST개

LLM이 topic을 새로 생성하는 게 아닌, 그래프에 존재하는 명시적인 후보 목록에서 선택. 또 하나만 강제로 고르지 않고 여러 후보를 선택해 누락 가능성을 줄임.
LLM 호출수: Topic 선택 1회, 선택된 Topic 별 Subtopic 선택: Nt회
핵심: 전체 말뭉치 -> 관련 Topic -> 관련 Subtopic 순서로 검색 공간을 좁힘

토픽 기반 그래프 탐색은 가장 관련도 높은 토픽 노드에서부터 탐색을 시작한다.
그리고 재귀적으로 관련된 서브토픽과 엔터티를 탐색한다. 트리플렛 후보를 검색하기 위해 그래프의 계층 구조를 활용하기 위해, 저자들은 LLM이 탐색을 이끄는 2단계 전략인 Topic-based Graph Traversal을 설계한다.

질문이 주어지면 LLM은 그래프에 존재하는 전체 토픽 노드 중에서 질문과 관련된 토픽 \(N*T\)개를 선택한다.
선택된 각 토픽에 대해, LLM은 그 토픽에 연결된 서브토픽 중 질문과 의미적으로 관련된 것 \(N*{ST}\)개를 선택한다.
실제로는 선택할 토픽 수 \(N*T\)와 토픽당 선택할 서브토픽 수 \(N*{ST}\)를 작은 값으로 제한한다. 그 결과 적은 수의 LLM 호출만으로도 탐색 방법을 대규모 말뭉치에 적용할 수 있다.

3. Query-based Retrieval
   3.2에서 선택한 Subtopic 아래의 Entity를 수집

쿼리기반 검색과 필터링, 이 단계에서 코사인 유사도는 쿼리와 각 후보 트리플렛의 엣지간 계산된다 그리고 가장 관련도 높은 정보는 정답 생성을 위한 최종 컨텍스트로 선택된다

핵심 아이디어 1줄:

실험에서 확인한 것: 기존 베이스라인 보다 높은 정확도와 안전성, 효율성을 유지,
실험 결과, TH-RAG는 개방형 생성 QA와 구체적 멀티홉 QA 벤치마크 모두에서 기존 graph-based RAG 방법들보다 우수한 성능을 보였다.

내가 가져가야할 점:
추가적인 ablation 연구는 계층적 그래프 설계와 검색 전략의 효과를 검증한다. 그 결과는 TH-RAG가 graph-based RAG 시스템에서 더욱 안정적인 멀티홉 추론을 가능하게 하는 유망하고 확장 가능한 접근법임을 보여준다.

LLM에 새 지식을 넣는 파인튜닝은 비싸고 망각 위험이 있다.
RAG가 대안이지만 일반 검색은 흩어진 정보의 멀티홉 추론에 약하다.
GraphRAG는 지식 그래프를 사용하지만, LLM이 만든 트리플 그래프는 파편화된다.
기존 그래프 보강은 계산 비용과 의미 왜곡 문제가 있다.
TH-RAG는 Topic → Subtopic → Triplet 계층으로 연결성을 보강한다.
관련 토픽부터 내려가며 후보 근거를 찾고, 질문과의 유사도로 최종 문맥을 고른다.
저자들은 이것이 정확성·안정성·확장성을 모두 개선한다고 주장한다.

## Method 3 정리 (2026-07-15)

### 전체 흐름

```text
문서 청크
→ Triplet + Subtopic + Topic 동시 추출
→ Topic–Subtopic–Entity 3계층 그래프 구축
→ 질문 관련 Topic 선택
→ 각 Topic에서 관련 Subtopic 선택
→ 선택된 Subtopic 아래의 Entity 전체 수집
→ 각 Entity의 1-hop 엣지와 원문 문장 수집
→ cosine similarity로 Top-K1 문장 선택
→ Top-K2 문장이 포함된 원본 chunk 확장
→ 문장 수준 + chunk 수준 context로 답변 생성
```

### 3.1 Hierarchical Graph Construction

- 각 청크에서 LLM 호출 한 번으로 Triplet, Subtopic, Topic을 동시에 추출한다.
- 노드는 Entity, Subtopic, Topic의 서로 겹치지 않는 세 집합으로 구성된다.
- 엣지는 Entity–Entity 관계, Subtopic–Entity 연결, Topic–Subtopic 연결의 세 종류다.
- 각 Entity는 트리플 관계 엣지와 Subtopic 계층 엣지를 가져 노드 고립 가능성을 낮춘다.
- 각 Entity–Entity 엣지에는 트리플이 추출된 원문 문장을 속성으로 저장한다.
- 원문 문장을 보존함으로써 추상화된 트리플만 사용할 때 발생하는 정보 손실을 줄이고 문장 수준 검색을 지원한다.
- 실험 설정에서는 문서를 1,200-token chunk로 나누고 100-token overlap을 사용한다.

### 3.2 Topic-based Graph Traversal

- LLM에 질문과 전체 Topic 후보 목록을 제공하고 관련 Topic `N_T`개를 선택한다.
- 선택된 각 Topic에 대해 질문과 Subtopic 후보 목록을 제공하고 관련 Subtopic `N_ST`개를 선택한다.
- 후보를 자유 생성하지 않고 그래프에 존재하는 명시적 목록에서 선택하여 모호성을 줄인다.
- 하나만 강제하지 않고 복수 후보를 허용하여 Topic/Subtopic 명칭 불일치로 인한 누락을 완화한다.
- LLM 호출 횟수는 Topic 선택 1회와 Subtopic 선택 `N_T`회로, 총 `1 + N_T`회다.
- Entity는 LLM이 별도로 선택하지 않는다. Algorithm 2와 부록 프롬프트 기준으로 선택된 Subtopic 아래의 Entity를 모두 수집한다.
- 본문의 “Topic, Subtopic, Entity 후보를 LLM에 제공한다”는 표현은 Algorithm 2 및 실제 Subtopic 프롬프트와 다소 일관되지 않는다.
- 본질적으로 관계 경로 추론이라기보다 대규모 말뭉치의 검색 범위를 계층적으로 좁히는 과정에 가깝다.

### 3.3 Query-based Retrieval

- 선택된 Subtopic에 연결된 Entity를 검색 시작점(anchor)으로 사용한다.
- 각 Entity의 1-hop 이웃을 탐색하고 연결된 Entity–Entity 엣지를 수집한다.
- 엣지 속성에 저장된 원문 문장들을 후보 근거 문장 집합으로 만든다.
- 질문과 후보 문장의 cosine similarity를 계산하고 관련성이 높은 Top-`K1` 문장을 1차 context로 선택한다.
- 그중 `K2`개 문장을 골라 해당 문장이 포함된 원본 chunk 전체를 추가한다. 기본 설정은 `K1=30`, `K2=5`다.
- 문장 수준 정보는 정밀한 근거를 제공하고, chunk 수준 정보는 주변 문맥과 그래프 구축 중 누락된 엔터티·근거를 보완한다.

### 3.4 Multi-hop Reasoning Robustness

- 3계층 연결로 파편화된 Entity 관계를 Topic/Subtopic 경로를 통해 우회할 수 있게 한다.
- 청크당 LLM 호출 한 번으로 계층을 구축하여 별도의 반복적 clustering·summarization 비용을 줄인다.
- 관련성이 높은 문장 주변의 context를 선별적으로 확장하여 엔터티 누락 위험을 줄인다.
- 엣지마다 원문 문장을 보존해 추상화된 Triplet 품질에만 의존하지 않고 문장 수준 근거로 돌아갈 수 있게 한다.
- 종합하면 검색 공간을 정답이 있을 가능성이 높은 영역으로 좁히고, 주변 문맥을 활용하며, 원본 말뭉치 정보 손실을 줄이는 것이 robustness의 근거다.

### Method를 읽으며 생긴 비판적 질문

- 전체 Topic 목록을 한 번에 입력하면 호출 횟수는 적어도 입력 토큰 비용과 latency가 커지지 않는가?
- 복수 Topic/Subtopic 선택은 recall을 높이지만 precision과 검색 비용을 악화시키지 않는가?
- LLM이 잘못된 Subtopic을 선택하면 해당 Entity에 도달하지 못한다. 이 오류를 후속 filtering이 복구할 수 있는가?
- 원문 문장 보존은 근거 손실을 줄이지만, 잘못된 graph construction·retrieval·generation으로 인한 환각까지 막지는 못한다.
